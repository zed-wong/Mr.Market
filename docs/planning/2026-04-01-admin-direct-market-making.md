# CEO Plan: Admin Direct Market-Making

Generated by /plan-ceo-review on 2026-04-01
Branch: main | Mode: SELECTIVE EXPANSION
Repo: zed-wong/Mr.Market

## Vision

### 10x Check
Mr.Market's market-making engine is fully built but has zero way to run it live. The Mixin deposit/withdrawal/exchange pipeline is incomplete (todo items 3-11). This plan adds a direct path: admin provides exchange + API key + pair + strategy, market making starts immediately. No Mixin. No payment flow. Just the engine.

This unlocks real-world validation of months of engineering work (pooled executors, tick coordination, fill routing, strategy controllers) without waiting for the full funding lifecycle.

## Architecture Decisions

| # | Decision | Choice | Reasoning |
|---|----------|--------|-----------|
| 1 | Queue vs direct call for start_mm | Direct call, extract shared method | Admin needs immediate error feedback |
| 2 | Distinguish direct orders | New `source` column on MarketMakingOrder | Clean, queryable, explicit |
| 3 | User-facing query filter | `WHERE source != 'admin_direct'` | Prevent admin test orders leaking to users |
| 6 | Test strategy | Unit tests + sandbox system spec | Prove the full path works end-to-end |

## Accepted Scope

### Phase 1: Backend (S-M)

**New files:**
- `server/src/modules/admin/market-making/admin-direct-mm.service.ts`
- `server/src/modules/admin/market-making/admin-direct-mm.controller.ts`
- `server/src/modules/admin/market-making/admin-direct-mm.dto.ts`
- Migration: add `source` column to MarketMakingOrder (default: `'payment_flow'`)

**Endpoints:**
1. `POST /admin/market-making/direct-start` — validate inputs, fetchBalance (advisory), resolveSnapshot, create MMOrder with source=admin_direct + state=created, call shared start method, update to state=running on success or state=failed on error
2. `POST /admin/market-making/direct-stop` — stop via extracted shared stop method
3. `GET /admin/market-making/direct-orders` — list WHERE source='admin_direct'
4. `GET /admin/market-making/direct-orders/:id/status` — read ExecutorRegistry + tracker state

**Shared extraction:**
- Extract core start_mm logic (processor L1070-1096) into shared method
- Extract core stop_mm logic into shared method
- Both queue handlers and direct-start call the shared methods

**Features:**
- Balance pre-check: fetchBalance(), warn if insufficient (advisory, not blocking)
- CCXT error mapping: `mapCCXTError(error): HttpException` helper catches ExchangeError/RateLimitExceeded/AuthenticationError/NetworkError, maps to descriptive BadRequestException
- Executor health check: status endpoint verifies executor responsiveness before returning 'active' state
- Async campaign join: `POST /campaign-join` returns immediately with status='pending', on-chain tx confirmed in background worker

**Entity change:**
- MarketMakingOrder: add `source: 'payment_flow' | 'admin_direct'` column (default 'payment_flow')
- User-facing queries: add `WHERE source != 'admin_direct'` filter

### Phase 2: Frontend (M)

**Note:** Use Svelte 4 syntax throughout (`export let`, `$:`, `on:click`, `<slot />`, `createEventDispatcher`). Matches existing admin pages and CLAUDE.md.

**All text strings must use `$_("i18n_key")` from svelte-i18n.** No plain English strings in components. Add keys to the appropriate en.json file.

**New route:** `interface/src/routes/(bottomNav)/(admin)/manage/market-making/direct/+page.svelte`

**Layout:** App UI, operator dashboard. Primary workspace (running orders) dominates. Secondary actions (form, campaign) are collapsible.

1. **Primary:** Running orders table (full width). Columns: exchange icon, pair, strategy, status badge (RUNNING=green, CREATED=yellow, STOPPED=gray, FAILED=red), created time, last tick, actions (Stop). Click row to open status drawer.
2. **Secondary:** Start form collapsed behind "Start Order" button. Expands inline below table. Fields: exchange dropdown, pair dropdown, strategy definition dropdown, config overrides (key-value table: Key | Value columns, Add Row button), Start button (2s debounce).
3. **Stop confirmation:** Clicking Stop opens a confirmation dialog: "Stop market making on [Pair]? This will cancel all open orders." [Cancel] [Stop] buttons.
3. **Detail:** Status detail in slide-over drawer on row click. Shows: open orders list, last tick timestamp, bid/ask spread, inventory balances. Polling every 5s. Shows "Last updated: Xs ago" with stale warning if >15s.
4. **Campaign panel:** Collapsible section below running orders. Tab-based: [Direct MM | Campaigns]. Campaigns tab: available campaigns list + Join button. Joined campaigns table with status badges (PENDING=yellow, JOINED=green, FAILED=red). Join opens a modal dialog with EVM address input + API key selector + Submit.
5. **Toasts:** svelte-sonner for success/error feedback.

**Design tokens (daisyUI + Tailwind, dark mode):**
- Page container: `bg-base-200`, `max-w-7xl mx-auto`
- Status badges: `badge badge-success` (running), `badge badge-warning` (created/pending), `badge badge-error` (failed), `badge` (stopped)
- Table: `table table-striped` matching existing admin patterns
- Buttons: `btn btn-primary` (Start/Join), `btn btn-error` (Stop), `btn btn-ghost` (back)
- Dropdowns: `select select-bordered`
- Modal: `modal modal-open`, `modal-box`
- Slide-over drawer: right-side panel, `fixed inset-y-0 right-0 w-96`
- Form inputs: `input input-bordered`
- Tab group: `tab tab-boxed`

**Responsive behavior:**
- Desktop primary (1024px+): full layout as designed
- Tablet (768-1023px): single-column layout, table scrolls horizontally
- Mobile (<768px): table scrolls horizontally, form stacks vertically, drawer takes full width
- Admin sidebar hidden by convention (no hamburger nav on admin routes)

**Accessibility:**
- Tab navigation through form fields, Enter to submit
- Focus trap inside modal dialog
- ARIA labels on icon-only buttons (Stop, back button)
- Touch targets minimum 44px
- Color not sole indicator of state (badges have text label, not just color)
- Screen reader: live regions for toast announcements

## Reviewer Concerns (resolved)

**Issue 1.1 — dryRun V1 is broken:** `executeIntents` is read once at `StrategyIntentExecutionService` construction from global config (L47). Setting `process.env.MARKET_MAKING_EXECUTE_INTENTS='false'` after boot has no effect on already-constructed services.
→ **Resolution:** dryRun removed from V1 scope. Deferred to TODOS.md as V2: thread dryRun flag through `StrategyOrderIntent.metadata` and check at L129.

**Issue 1.2 — userId for admin orders:** MarketMakingOrder requires `userId`.
→ **Resolution:** Use the admin's JWT identity (extracted from the auth token) as `userId`. If no Mixin user exists, use a sentinel value `'admin-direct'`. `rewardAddress` is nullable, set to null.

**Issue 1.3 — Required entity fields (bidSpread, askSpread, etc.):** These come from `mapStrategySnapshotToMarketMakingOrderFields()` which extracts them from `resolvedConfig`.
→ **Resolution:** Already solved. The strategy definition's `defaultConfig` merged with `configOverrides` produces `resolvedConfig`, and `mapStrategySnapshotToMarketMakingOrderFields()` maps those to the required columns with sensible defaults. Same code path as payment flow.

**Issue 1.4 — API key / account selection:** Admin must specify which exchange account to use.
→ **Resolution:** Add `accountLabel` field to the DTO. This is already the pattern used by `ExchangeInitService.getExchange(exchangeName, accountLabel)` and the existing admin deposit-address endpoint.

**Issue 2.1 — State transition duplication:** Shared method must not re-set state=running.
→ **Resolution:** Shared method starts at L1070 (after state transition). Caller (direct-start or queue handler) handles its own state transition before calling the shared method.

**Issue 2.2 — Sandbox is system-wide boot mode:** `ExchangeInitService` sandbox is controlled by env vars at boot, not per-request.
→ **Resolution:** For V1, sandbox toggle is removed from the per-order DTO. Instead, document that admin should deploy with sandbox env vars set to test on testnet. For V2 (future), lazy-init a sandbox CCXT instance in the admin-direct service. Re-estimated: V1=removed from scope, V2=M effort deferred to TODOS.md.

## Key Design Principles

1. Creates real MarketMakingOrder with source=admin_direct (not separate entity)
2. Reuses 100% of execution infrastructure unchanged
3. Direct synchronous call (not queue) for immediate admin feedback
4. Balance pre-check is advisory only (warns, doesn't block)
5. dryRun V1: system-wide env toggle; V2 (future): per-order flag in intent metadata
6. Sandbox V1: deploy with sandbox env vars; V2 (future): per-order sandbox init
7. userId for admin orders: admin JWT identity or sentinel 'admin-direct'
8. Required entity fields populated via existing mapStrategySnapshotToMarketMakingOrderFields()
9. accountLabel in DTO to select which configured exchange API key to use

### Phase 4: HuFi Campaign Join (M)

**New binding model:**
```
  EVM Address (wallet)  ──────┐
                              ├── scoped join ──▶ HuFi Campaign (on-chain)
  API Key (exchange acct) ────┘

  MarketMakingOrder ──binds to──▶ API Key (accountLabel)
  API Key ──binds to──▶ Joined Campaigns
  EVM Address ──used for──▶ On-chain contract calls (nonce, signing, auth)

  Uniqueness: (evmAddress, apiKeyId, campaignAddress, chainId) = at most 1 join
```

**New entity: `CampaignJoin`**
```typescript
@Entity()
export class CampaignJoin {
  @PrimaryGeneratedColumn('uuid')
  id: string;

  @Column()
  evmAddress: string;        // wallet that signed the on-chain join

  @Column()
  apiKeyId: string;          // which exchange API key account

  @Column()
  chainId: number;           // campaign chain

  @Column()
  campaignAddress: string;   // campaign contract address

  @Column({ nullable: true })
  orderId: string;           // linked MarketMakingOrder (if any)

  @Column()
  status: string;            // 'pending' | 'linked' | 'detached' | 'joined' | 'failed'
                            // pending: join tx sent, awaiting confirmation
                            // linked: join confirmed, has running MM order
                            // detached: join confirmed, MM order stopped
                            // joined: on-chain join complete
                            // failed: join tx failed

  @CreateDateColumn()
  createdAt: Date;

  @UpdateDateColumn()
  updatedAt: Date;

  // Unique constraint: (evmAddress, apiKeyId, campaignAddress, chainId)
}
```

**Entity change: `MarketMakingOrder`**
- Add `apiKeyId` column (nullable) — binds direct-start orders to specific API key

**Admin endpoints:**
- `GET /admin/market-making/campaigns` — list available HuFi campaigns (from CampaignService.getCampaigns)
- `POST /admin/market-making/campaign-join` — join a campaign with {evmAddress, apiKeyId, chainId, campaignAddress}
  1. Check uniqueness constraint (evmAddress + apiKeyId + campaign)
  2. Call CampaignService.joinCampaignWithAuth(evmAddress, privateKey, chainId, campaignAddress)
  3. Create CampaignJoin record
  4. Optionally link to a running MarketMakingOrder if one exists for this apiKeyId+pair
- `GET /admin/market-making/campaign-joins` — list joins for an API key
- `POST /admin/market-making/direct-start` — updated to accept optional campaignJoinId, link order to campaign join

**Admin frontend additions:**
- Campaign list panel on the direct MM page
- "Join Campaign" button with EVM address input + API key selector
- Joined campaigns table showing (evmAddress, apiKey, campaign, status)
- When starting direct MM, option to link to an existing campaign join

**EVM private key handling:**
- Uses existing `WEB3_PRIVATE_KEY` env var via `web3.private_key` config (already in Web3Service)
- EVM wallet address derived from the private key at runtime
- Key is never persisted beyond env, never logged
- No new key management needed
