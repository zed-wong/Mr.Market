## Vision

### 10x Check
Mr.Market becomes the "Docker of market making" — a single binary you configure with a YAML file, run `mr-market start`, and it's live on your exchange. No database setup, no Redis, no Mixin. Pure runtime. One operator, one config, one command.

### Platonic Ideal
The ideal experience:
1. User downloads `mr-market` binary (or runs `bunx @mr-market/cli`)
2. Creates `config.yaml` with exchange credentials and strategy params
3. Runs `mr-market start`
4. Opens `localhost:3000` to see live dashboard
5. Hot-edits config to adjust spread, sees immediate effect
6. All strategies available from day one (pure MM, inventory skew, arbitrage, volume)

## Accepted Scope

### Phase 1: Core Infrastructure (S)
1. **MODE environment variable** — `single` vs `multi` toggles module boot
2. **Conditional module loading** — Mixin modules skip boot in single mode
3. **Zero-infra single mode** — In-memory state, no SQLite/Redis required
4. **YAML config loader** — Load exchange credentials, pairs, strategy config

### Phase 2: CLI Entry Point (S)
5. **New /packages/cli** — Standalone CLI with `mr-market start`
6. **Bun compile target** — Single-binary distribution
7. **bunx support** — `bunx @mr-market/cli start --config config.yaml`

### Phase 3: User Experience (M)
8. **Built-in dashboard** — Web UI at localhost:3000 (orderbook, orders, PnL, ticks)
9. **DRY_RUN mode** — Log actions without exchange execution
10. **Hot-reload config** — Watch file, apply changes without restart

### Phase 4: Multi-Pair & All Strategies (M)
11. **Multi-pair config** — One config, multiple pairs, shared connection pool
12. **All strategies enabled** — Pure MM, inventory skew, arbitrage, volume

## Key Clarifications (from review)

1. **CLI↔Server relationship** — CLI imports `server/` directly and starts NestJS with conditional module loading. Boots only runtime modules (strategy, execution, trackers, tick).

2. **Zero-infra semantics** — Single mode eliminates Redis (Bull queues) but defaults to in-memory state. SQLite optional persistence, not default.

3. **Sandbox vs DRY_RUN** — `sandbox: true` uses exchange testnet API. `dry_run: true` logs without executing. Can be combined.

4. **Queue elimination** — Single mode runs tick coordination synchronously in-process. `TickModule` and `ClockTickCoordinator` work without modification.

5. **Hot-reload design** — Watches config.yaml, validates, updates strategy params on next tick cycle. No mid-tick changes.

6. **Strategy types mapping** — `pureMarketMaking` → `PureMarketMakingStrategyController`, `arbitrage` → `ArbitrageStrategyController`, `volume` → `VolumeStrategyController`. Inventory skew is a parameter of pureMarketMaking.

## Config Schema (Draft)

```yaml
# config.yaml - Single-operator mode configuration

mode: single  # single | multi (future)

# Exchange credentials
exchange:
  name: binance
  api_key: ${BINANCE_API_KEY}
  api_secret: ${BINANCE_API_SECRET}
  sandbox: false  # Set true for testnet

# Pairs and strategies
pairs:
  - symbol: BTC/USDT
    strategy:
      type: pureMarketMaking
      spread_bps: 10
      order_amount: 0.01
      min_spread_bps: 5
      max_spread_bps: 50
      order_levels: 1
      inventory_skew_base: 0  # 0 = disabled
  - symbol: ETH/USDT
    strategy:
      type: pureMarketMaking
      spread_bps: 15
      order_amount: 0.1

# Runtime settings
runtime:
  tick_interval_ms: 1000
  dry_run: false  # Set true to log without executing

# Dashboard (optional)
dashboard:
  enabled: true
  port: 3000

# Logging
logging:
  level: info  # debug | info | warn | error
  format: json  # json | pretty
```

## Open Engineering Questions (NOT RESOLVED)

These need decisions before implementation:

1. **StrategyRuntimeDispatcherService DB-coupling** — `start_mm` loads `MarketMakingOrder` from DB to extract controllerType+config. In single-mode there's no DB-backed order. Options: (A) Create `SingleModeStrategyLauncher` that calls `startByStrategyType()` directly, or (B) Use in-memory `MarketMakingOrder` substitute.

2. **ExchangeInitService DB-coupling** — Reads `APIKeysConfig` DB entity. Single-mode needs config-from-YAML instead.

3. **Ledger/Durability dependency chain** — `StrategyModule` transitively imports `LedgerModule` and `DurabilityModule`. Single-mode needs to cut this dependency.

## NOT in Scope
- Runtime package extraction (@mr-market/runtime) — deferred, thin CLI wrapper sufficient
- Full paper-trading mode — DRY_RUN covers basic simulation
- Cross-pair arbitrage with complex routing — single-pair arbitrage sufficient at launch
- TEE integration — future work for trust-minimized execution
- Multi-tenant SaaS changes — this plan focuses on single-operator mode only
